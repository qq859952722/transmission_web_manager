### 2.1 torrent_get: `bytes_completed`

| 属性 | 值 |
|---|---|
| 旧协议字段名 | `bytesCompleted` |
| 4.1.x 字段名 | `bytes_completed` |
| 类型 | array (number[]) |
| 官方定义 | 各文件已完成字节数组，长度等于文件数量 |
| **本地实测** | ✅ 真实存在，返回 `[2992299819]` 等数值 |
| 与已有功能的关系 | `files[].bytesCompleted` 已在文件列表中单独返回每个文件的完成字节数；`fileStats[].bytesCompleted` 同样返回此值 |

**分析**: 此字段是 `files` 和 `fileStats` 子对象中 `bytesCompleted` 的顶层汇总版本，作为独立数组返回。项目已通过 `t.files[i].bytes_completed` 在文件列表中逐个显示每个文件的完成字节数（[ui-detail.js](file:///home/qq/transmission_web_manager/js/ui-detail.js#L122) 的 `_renderFiles` 中 `pct = file.length > 0 ? (file.bytes_completed / file.length * 100).toFixed(1) : '0.0'`）。

**UI 实现建议**: ❌ **无需实现**。功能已通过 `files[].bytes_completed` 完全覆盖，顶层汇总数组仅是冗余的另一种数据组织形式，无额外信息价值。

### 2.2 torrent_get: `webseeds_ex`

| 属性 | 值 |
|---|---|
| 旧协议字段名 | 无（JSON-RPC 2.0 新增） |
| 4.1.x 字段名 | `webseeds_ex` |
| 类型 | array of objects |
| 官方定义 | Web 种子扩展信息，包含 `url`、`is_downloading`、`download_bytes_per_second` |
| **本地实测** | ⚠️ 当前实例无 Web 种子种子，无法实测返回值。但官方文档明确此字段为 4.2.0+ 新增 |
| 与已有功能的关系 | `webseeds` (旧字段，返回 string[]) 已在 DETAIL_FIELDS 中请求并在详情页显示 |

**子字段详情**:

| 子字段 | 类型 | 说明 | 本地实测 |
|---|---|---|---|
| `url` | string | Web 种子 URL | 与 `webseeds[]` 中的 URL 相同 |
| `is_downloading` | boolean | 是否正在从此 Web 种子下载 | 未实测 |
| `download_bytes_per_second` | number | 从此 Web 种子的下载速度 (B/s) | 未实测 |

**分析**: `webseeds_ex` 是 `webseeds` 的扩展版本，额外提供了每个 Web 种子的实时下载状态和速度。`webseeds` 已被官方标记为 DEPRECATED（4.2.0），未来将被 `webseeds_ex` 替代。但当前 4.1.x 版本中 `webseeds_ex` 尚未正式可用。

**UI 实现建议**: ⏳ **暂缓实现**。当前 4.1.x 版本此字段尚未稳定，建议在 4.2.0 正式发布后实现。实现时可替换现有 `webseeds` 显示，在 Web 种子列表中增加下载状态和速度列。实现位置：[ui-detail.js](file:///home/qq/transmission_web_manager/js/ui-detail.js) 的 Web 种子显示区域。

### 2.3 torrent_get: `sequential_download_from_piece`

| 属性 | 值 |
|---|---|
| 旧协议字段名 | `sequential_download_from_piece` |
| 4.1.x 字段名 | `sequential_download_from_piece` |
| 类型 | number |
| 官方定义 | 顺序下载起始分片索引，配合 `sequential_download` 使用 |
| **本地实测** | ✅ 真实存在，默认返回 `0`。设置后可正确读取（实测设置 `5` 后读取返回 `5`） |

**测试过程**:
```bash
# 设置顺序下载 + 起始分片
curl -d '{"method":"torrent-set","arguments":{"ids":[3],"sequential_download":true,"sequential_download_from_piece":5}}'
# 读取验证
curl -d '{"method":"torrent-get","arguments":{"ids":[3],"fields":["sequential_download","sequential_download_from_piece"]}}'
# 返回: {"sequential_download":true,"sequential_download_from_piece":5}
```

**分析**: 此字段允许用户指定从哪个分片开始顺序下载，适用于需要优先下载文件中间部分的场景（如视频文件从特定位置开始播放）。项目已实现 `sequential_download` 开关（[ui-detail-extras.js](file:///home/qq/transmission_web_manager/js/ui-detail-extras.js) 的 `renderSettings`），但未实现起始分片设置。

**UI 实现建议**: 🟡 **低优先级**。可在设置 Tab 的顺序下载开关旁增加一个"起始分片"输入框，仅在 `sequential_download` 为 `true` 时显示。实现方案：
1. 在 [ui-detail-extras.js](file:///home/qq/transmission_web_manager/js/ui-detail-extras.js) 的 `renderSettings` 中增加输入框
2. 在 [rpc.js](file:///home/qq/transmission_web_manager/js/rpc.js) 的 `_torrentSetKeyMap` 中已有 `sequential_download` 映射，需添加 `sequential_download_from_piece` 映射

### 2.5 tracker_stats 子对象: 刮取相关字段

以下字段均属于 `tracker_stats` 子对象，与 Tracker 的 scrape（刮取）操作相关：

| 字段 | 旧协议名 | 类型 | 说明 | 本地实测 |
|---|---|---|---|---|
| `scrape` | `scrape` | string | 刮取 URL | ✅ 返回 `udp://tracker.opentrackr.org:1337/scrape` |
| `scrape_state` | `scrapeState` | number | 刮取状态 | ✅ 返回 `1` |
| `has_scraped` | `hasScraped` | boolean | 是否已刮取 | ✅ 返回 `true` |
| `last_scrape_result` | `lastScrapeResult` | string | 上次刮取结果 | ✅ 返回 `"IPv4 connection failed"` 等 |
| `last_scrape_start_time` | `lastScrapeStartTime` | number | 上次刮取开始时间 | ✅ 返回 `0` |
| `last_scrape_succeeded` | `lastScrapeSucceeded` | boolean | 上次刮取是否成功 | ✅ 返回 `false` |
| `last_scrape_time` | `lastScrapeTime` | number | 上次刮取时间 | ✅ 返回 Unix 时间戳 |
| `last_scrape_timed_out` | `lastScrapeTimedOut` | boolean | 上次刮取是否超时 | ✅ 返回 `true` |
| `next_scrape_time` | `nextScrapeTime` | number | 下次刮取时间 | ✅ 返回 Unix 时间戳 |

**分析**: 当前 Tracker Tab（[ui-detail.js](file:///home/qq/transmission_web_manager/js/ui-detail.js#L270) 的 `_renderTrackers`）仅显示了 announce 相关信息（状态、上次/下次宣告时间、宣告结果），未显示 scrape 相关信息。scrape 信息对于诊断 Tracker 连接问题有参考价值。

**与已有功能的关系**: `_renderTrackers` 已显示 `announce_state`、`last_announce_time`、`next_announce_time`、`last_announce_result`、`last_announce_succeeded` 等 announce 字段。scrape 字段是与之平行的另一套状态信息。

**UI 实现建议**: 🟡 **中低优先级**。可在 Tracker Tab 中增加"刮取状态"列或在 Tracker 详情弹窗中显示。实现方案：
1. 在 Tracker 表格中增加"刮取状态"和"上次刮取"列
2. 或参考 Peer 详情弹窗模式，为每个 Tracker 增加详情按钮，在弹窗中显示完整的 announce + scrape 信息

**UI 实现建议**: 🟡 **低优先级**。当前硬编码的单位换算已能正常工作，且 Transmission 的单位体系不太可能频繁变更。如需实现，方案为：

2. 在 [utils.js](file:///home/qq/transmission_web_manager/js/utils.js) 中将 `formatBytes`/`formatSpeed` 改为从 session 数据读取单位配置
3. 缓存 session 中的 units 配置，避免每次格式化都查找

### 2.9 torrent_set: `tracker_list`

| **本地实测** | ✅ 真实可用。设置和读取均正常 |

**测试过程**:

**分析**: `tracker_list` 是 `tracker_add`/`tracker_remove`/`tracker_replace` 的替代方案（后三者已被标记为 DEPRECATED）。它使用单个字符串一次性设置所有 Tracker，支持 tier 分组（空行分隔）。当前项目使用 `trackerAdd`/`trackerRemove`/`trackerReplace` 实现 Tracker 的增删改操作（[ui-dialog.js](file:///home/qq/transmission_web_manager/js/ui-dialog.js) 的 `showAddTracker`/`showReplaceTracker`）。

**UI 实现建议**: 🟢 **建议实现**。`tracker_add`/`tracker_remove`/`tracker_replace` 已被官方标记为 DEPRECATED，未来版本可能移除。实现方案：
1. 在 [rpc.js](file:///home/qq/transmission_web_manager/js/rpc.js) 的 `_torrentSetKeyMap` 中添加 `trackerList: 'tracker_list'` 映射
2. 修改 [ui-dialog.js](file:///home/qq/transmission_web_manager/js/ui-dialog.js) 的 `showAddTracker`，将新增 Tracker 追加到现有 `trackerList` 字符串末尾
3. 修改 `showReplaceTracker`，使用 `tracker_list` 一次性替换所有 Tracker
4. 在 DETAIL_FIELDS 中添加 `trackerList` 字段请求（已标记为"数据已请求"）
